# 아키텍처 변경 의사결정 기록

## 최초 목표

이미지를 업로드하면 사진 속 모든 객체를 자동으로 감지하고, 각 영역을 그 객체의 이름(글자)으로 채우는 앱을 만드는 것이었다. 사용자가 모드를 선택하거나 설정을 건드릴 필요 없이, 어떤 사진이든 넣으면 알아서 처리되는 경험을 목표로 했다.

---

## 왜 단일 모드로는 불가능했는가

### 제약 1 — 브라우저 실행 가능 모델의 클래스 제한

가장 먼저 검토한 모델은 YOLO-seg였다. COCO 데이터셋 기반으로 학습된 이 모델은 브라우저에서 Transformers.js로 실행할 수 있고, bounding box와 segmentation mask를 동시에 제공한다는 점에서 이상적인 후보였다.

그러나 COCO 80개 클래스라는 한계가 있다. 사람 사진을 넣으면 `person` 하나로 전체를 하나의 덩어리로만 잡는다. 눈, 코, 입, 피부, 머리카락 같은 신체 부위는 COCO에 클래스로 정의되어 있지 않아 인식 자체가 불가능하다. 풍경 사진에서도 키캡, 피규어, 악기처럼 COCO에 없는 사물은 감지가 안 된다. "거의 모든 객체를 인식하고 싶다"는 목표와 정면으로 충돌했다.

### 제약 2 — 브라우저 실행 가능한 오픈 어휘 모델의 한계

클래스 제한 없는 오픈 어휘 모델들을 검토했다.

| 모델 | 문제점 |
|------|--------|
| Florence-2 | 브라우저 실행 가능, 오픈 어휘이지만 segmentation mask 없이 bounding box만 반환 |
| YOLO-World | 오픈 어휘 + YOLO 속도지만 Transformers.js 지원 및 mask 출력 여부 미검증 |
| SAM2 | mask 품질 최고지만 모델 크기 300MB+, 브라우저 실행 사실상 불가 |
| Grounding DINO + SAM2 | 두 모델 합산 수백 MB, 추론 속도 브라우저 한계 초과 |

브라우저에서 실행 가능하면서 오픈 어휘 감지와 픽셀 단위 마스크를 동시에 제공하는 모델이 현재 존재하지 않는다는 결론에 도달했다.

### 제약 3 — 서버를 허용하면 비용이 발생한다

서버 경유를 허용하면 SAM2 + VLM API 조합으로 최고 품질을 낼 수 있다. 그러나 SAM2를 GPU 서버에서 돌리려면 Replicate 같은 서비스를 써야 하고, 이미지당 $0.01~0.05의 비용이 발생한다. 1인 개발 프로젝트에서 트래픽에 비례해 비용이 나가는 구조는 적합하지 않았다.

## 어떻게 인물/풍경 이중 모드로 귀결됐는가

제약들을 정리하고 나니, 사진의 성격에 따라 요구사항이 근본적으로 다르다는 점이 보였다.

**인물 사진**은 사람이 주 피사체이고, 사용자가 기대하는 결과는 "눈은 눈이라는 글자, 피부는 피부라는 글자"처럼 신체 부위 단위의 정밀한 인식이다. 이 요구를 위해서는 MediaPipe가 이상적이다. Google이 만든 공식 JS SDK가 있고, Face Mesh(468개 랜드마크)와 Selfie Segmentation으로 신체 부위를 픽셀 단위로 정밀하게 분리할 수 있다. 브라우저에서 실행되고, 모델 크기도 ~6MB로 가볍다.

**풍경 사진**은 다양한 사물이 복합적으로 있고, 사용자가 기대하는 결과는 "키캡은 키캡, 커피잔은 커피잔"처럼 가능한 많은 사물을 인식하는 것이다. 이 요구를 위해서는 오픈 어휘 모델이 필요한데, 브라우저 실행 가능한 옵션 중 가장 현실적인 선택은 Gemini 2.0 Flash다. 무료 티어(일 1,500회)가 넉넉해서 1인 개발 MVP에서 사실상 무과금이고, 인식 품질이 매우 높다. 단, 마스크 없이 bounding box만 반환하는 것이 유일한 타협점이다.

또한 모드를 나누는 것이 단순히 기술적 제약의 결과만은 아니었다. 인물 사진과 풍경 사진은 사용자가 기대하는 결과물의 미감 자체가 다르다. 인물 사진에서는 신체 부위가 정밀하게 구분되는 것이 흥미롭고, 풍경 사진에서는 다양한 사물이 많이 잡히는 것이 흥미롭다. 모드를 나누는 것이 UX 측면에서도 더 재미있는 경험을 만들 수 있다고 판단했다.

## 최종 결정

### 인물 모드

| 항목 | 선택 |
|------|------|
| 모델 | MediaPipe Face Mesh + Selfie Segmentation |
| 실행 위치 | 브라우저 (서버 불필요) |
| 마스크 | 픽셀 단위 정밀 마스크 |
| 인식 범위 | 눈, 눈썹, 코, 입, 귀, 얼굴 윤곽, 피부, 머리카락 |
| 비용 | 완전 무료 |

### 풍경 모드

| 항목 | 선택 |
|------|------|
| 모델 | Gemini 2.0 Flash Vision API |
| 실행 위치 | Vercel API Route (서버) |
| 마스크 | Bounding box (픽셀 마스크 없음) |
| 인식 범위 | 오픈 어휘 — 클래스 제한 없음 |
| 비용 | Gemini 무료 티어 (일 1,500회) — MVP 트래픽에서 사실상 무과금 |

### 모드별 렌더링 전략

두 모드는 입력 데이터 형태가 달라 렌더링 전략도 분리된다.

- 인물 모드: 픽셀 마스크를 샘플링해 마스크 내부 좌표에 글자를 반복 배치
- 풍경 모드: bounding box 영역 안에 글자를 반복 배치

## 타입 시스템 설계

두 모드의 결과 구조가 완전히 달라 별도 타입으로 분리하고, Discriminated Union으로 묶었다.

```ts
type AnalysisResult =
  | { mode: 'portrait'; data: PortraitAnalysisResult }
  | { mode: 'scene'; data: SceneAnalysisResult }
```

`mode` 필드로 TypeScript가 자동으로 타입을 좁혀주므로, `result.mode === 'portrait'`인 블록에서만 `bodyPartDetections`에 접근할 수 있고, 잘못된 접근은 컴파일 오류로 잡힌다.
